# UI/UX Audit — Bug & Polish Backlog

Twenty-nine issues from a UI/UX audit of Arsenal Ops, grouped into **9
area-based tickets** — one per screen/component cluster, each sized to a
single engineer's PR. Every area ticket carries a checklist of its
constituent issues with current-vs-expected behavior and a **Location**
pointer (file + line) so the code is one click away. Line numbers are from
the audit-time working tree and may drift — treat them as starting points.

Each area mixes Bug and UI Polish work; the ticket's **Priority** reflects
its highest-impact item, and every issue line is tagged individually. The
original audit item number is shown in brackets (e.g. `[5]`) for traceability.

the rest. Everything else is frontend-only and independent.

## Ticket A — Dashboard (Home) polish & load-state fixes

**Type:** Bug + UI Polish
**Priority:** High
**Screen/Component:** Dashboard (Home) — capacity card, project navigation, My Tasks list

Four issues on the landing page, from a blank-on-load data bug down to badge polish.

### Issues

- **[1] (Bug, High) Capacity-this-week card renders blank until user interacts.**
  - *Current:* On first load the card is empty; the figure (e.g. `48h / 40h`) only appears after an unrelated interaction — a race / missing re-render.
  - *Expected:* Loading skeleton while the query is in flight, then the figure populates on its own.
  - *Location:* `app/src/components/ProjectsPage/MyCapacityCard/index.tsx` — `useQuery(['myCapacity'])` (~L17–24), `retry` returns `false` only on 404 (~L20–22); `app/src/components/ProjectsPage/MyCapacityCard/components/CapacityTile.tsx` loading/render branch (~L31–37); rendered by `app/src/components/ProjectsPage/DashboardStats.tsx` (~L113).

- **[2] (Bug, Medium) No loading indicator when navigating into a project.**
  - *Current:* Clicking a project's arrow changes the URL but the old dashboard stays for 1–2s with no feedback, so the click looks dead.
  - *Expected:* A route spinner / project-detail skeleton replaces the old content as soon as navigation starts.
  - *Location:* `app/src/components/ProjectsPage/ProjectsBox.tsx` — `onClick={() => onProjectClick(project.id)}` (~L92); `app/src/pages/ProjectsPage/ProjectsPage.tsx` — `navigate(\`/project/${id}\`)` (~L123); `app/src/pages/ProjectDetail/ProjectDetail.tsx` — skeleton only while `isLoading` (~L160–161); `app/src/App.tsx` — route `Suspense`/`RouteSpinner` fires only before the lazy chunk is cached (~L187).

- **[3] (UI Polish, Low) My Tasks project badges truncate with no tooltip.**
  - *Current:* Long names clip (e.g. `Big Idea and Bl...`) with no hover affordance.
  - *Expected:* Hover reveals the full name (native `title` or shadcn/ui Tooltip).
  - *Location:* `app/src/components/ProjectsPage/MyTasksBox/components/WorkItemRow.tsx` — badge `<span className="… truncate min-w-0">` (~L34), no `title`.

- **[4] (UI Polish, Low) Over-capacity figure shown red with no context/breakdown.**
  - *Current:* Over capacity the figure turns red (`#EF4444`) with no breakdown or link; the detail-modal open is gated on `!isLoading && data`.
  - *Expected:* Over-capacity state offers an inline breakdown, tooltip, or reliable click-through to explain the overload.
  - *Location:* `app/src/components/ProjectsPage/MyCapacityCard/index.tsx` — `status`/`statusColor` when `remaining <= 0` (~L31–34), modal open (~L67); `.../components/CapacityTile.tsx` figure `style={{ color: statusColor }}` (~L35–37).

### Acceptance Criteria
- [ ] Capacity card shows a skeleton, then populates without user interaction.
- [ ] Navigating into a project shows an immediate loading state.
- [ ] Truncated project badges reveal their full name on hover.
- [ ] Over-capacity capacity figure provides some path to context (breakdown/tooltip/link).

## Ticket D — Project Manager tab: hours don't reconcile

**Type:** Bug
**Priority:** High
**Screen/Component:** Project Detail → Project Manager tab → hours summary cards

### Issue

- **[13]** *Current:* Total / Logged / Remaining don't add up (e.g. 37 / 17 / 18 — should be 20). Remaining is computed per-item as `max(0, estimated − logged)` and excludes DONE items, so it diverges from `total − logged`.
  - *Expected:* The three values reconcile under one documented rule (decide whether DONE items and the `max(0, …)` floor are included), and the cards reflect it.
  - *Repro:* Open a project with logged hours → Project Manager tab → compare the three figures.
  - *Location:* Backend `backend/routers/workitems.py` — `get_hours_analytics()` (~L2503), `total_remaining = sum(max(0, estimated − logged) for non-DONE items)` (~L2977–2981). Frontend `app/src/components/PMView/sections/PMSummaryCards.tsx` — total/logged/remaining (~L21, L35, L49). Endpoint `/api/workitems/projects/{project_id}/hours-analytics`.

### Acceptance Criteria
- [ ] Total, Logged, and Remaining reconcile arithmetically under a documented rule.
- [ ] Rule for DONE items and negative-remaining clamping is explicit and consistent between backend and cards.
